# AgoraX 产品需求文档 (PRD)

> **版本**: v1.0
> **日期**: 2026-07-07
> **定位**: 智能体-人类混合会议的协议管道层（Protocol Router）
> **核心理念**: 我们不做 Agent，我们做 AIM (Agent-Inclusive Meeting)。只提供管道，Agent 的脑子留在用户自己的 Runtime 里。

---

## 1. 产品概述

AgoraX 是一个面向 Agent 与人类混合会议的**实时协议管道系统**。它不定义 Agent 的行为逻辑，不提供预设的 AI 角色，不强制用户使用特定的 Agent 框架——它只做一件事：**让任何 Agent Runtime（Claude Code、Codex、Python 脚本、Shell、浏览器）都能通过一行命令接入线上会议，与人类或其他 Agent 进行实时语音/文字交流。**

类比：
- 如果说传统视频会议是"Outlook"（封闭生态），AgoraX 就是"AIM"（开放协议）。
- 如果说 AutoGPT 是"Agent 花园"（把 Agent 种进我们的系统），AgoraX 就是"协议管道"（任何 Agent 都能连进来）。

---

## 2. 核心定位

| 维度 | 传统视频会议 | 传统 AI 会议产品 | **AgoraX** |
|------|------------|----------------|-----------|
| **Agent 接入** | 不支持 | 提供预设 AI 角色，用户选择 | **任何 Agent Runtime 都能接入** |
| **SDK 依赖** | 必须安装 SDK | 必须安装 SDK | **零 SDK，curl 即可** |
| **行为控制** | 人类主导 | 平台定义 AI 行为 | **Agent 自己决定怎么回应** |
| **音频处理** | 平台混音 | 平台 TTS | **Agent 可自带 TTS 或调用云服务** |
| **协议开放** | 封闭 | 半封闭 | **完全开放 JSON Lines 协议** |

## 3. 目标用户

### 3.1 主要用户

1. **AI 研究者 / 开发者**
   - 已有自己的 Agent 系统（基于 Claude Code、Codex、自研 LLM）
   - 希望让 Agent 参加学术会议、技术分享、圆桌讨论
   - 痛点：现有平台要求把 Agent"种"进他们的系统，改造成本高

2. **会议组织者**
   - 举办线上技术大会、学术研讨会
   - 希望邀请人类专家 + AI 专家共同参与
   - 痛点：现有平台不支持 Agent 以平等身份参会

3. **企业技术团队**
   - 内部技术分享、Code Review 会议
   - 希望让 AI 助手作为"同事"参与讨论
   - 痛点：现有会议工具把 AI 当作"辅助功能"而非"参会者"

### 3.2 用户画像

** persona 1: 田博士**
- 身份：AI 实验室负责人
- 场景：每周举办"AI 前沿讨论会"，邀请 3 位人类研究员 + 2 个 Agent（Claude Code 驱动的研究助手）
- 痛点：Agent 只能以"文字聊天机器人"形式存在，无法参与语音圆桌讨论
- 需求：Agent 能以语音身份发言、打断、提问，与人类平等交流

** persona 2: 开源社区组织者**
- 身份：开源项目维护者
- 场景：举办线上技术分享，演讲者是人类，但希望有一个 Agent 实时回答观众问题
- 痛点：现有平台要求把问答逻辑写进平台，无法用自己训练的模型
- 需求：Agent 通过 Webhook 接收问题，用自己的模型回答，通过语音返回

## 4. 用户场景

### 4.1 场景一：学术 Keynote + Agent 共演（核心场景）

**流程：**
1. 人类演讲者分享论文，控制 Slides 翻页
2. 演讲者说："关于这个实验结果，我的 Agent 有一些补充分析"
3. Agent 通过语音接入："我重新跑了实验，发现当学习率大于 0.001 时..."
4. 观众提问，Agent 通过 RAG 检索论文后回答

**价值：** 人类专家 + Agent 助手形成"1+1>2"的演讲效果，Agent 可以 24 小时值守 Poster Session。

### 4.2 场景二：混合圆桌讨论（Panel Discussion）

**流程：**
1. 3 位人类专家 + 2 个 Agent 围坐（虚拟圆桌）
2. 人类 A 提出观点
3. Agent B（基于 Codex）通过语音反驳："我不同意，因为代码实现上..."
4. Agent C（基于自研模型）补充："我补充一个数据点..."
5. 人类 D 总结

**价值：** Agent 可以基于不同知识库（代码、论文、数据）提供多元视角，讨论不受人类专家时间限制。

### 4.3 场景三：24 小时 Poster Session

**流程：**
1. 人类作者上传论文，生成 Agent 分身（基于论文内容 + 作者声线克隆）
2. 观众随时进入虚拟展厅，与 Agent 对话
3. Agent 回答技术细节，遇到超纲问题标记给人类作者
4. 人类作者上线后，Agent 交接上下文

**价值：** 解决学术会议"作者必须在场"的时间限制，全球时区差异不再是问题。

### 4.4 场景四：企业内部技术分享

**流程：**
1. 工程师分享代码架构设计
2. 团队里的"Code Review Agent"（基于 Claude Code）实时监听
3. Agent 在适当时候插话："这里有一个潜在的竞态条件..."
4. 会议结束后，Agent 自动生成纪要 + Action Items

**价值：** Agent 成为团队的"常驻成员"，参与日常技术讨论。

## 5. 功能模块

### 5.1 核心功能（必须实现）

| 模块 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| **协议管道** | WebSocket/HTTP/SSE 消息路由 | P0 | 核心能力，所有消息的中转站 |
| **房间管理** | 创建/加入/离开房间 | P0 | 支持多房间并发 |
| **身份认证** | DID + API Key + 签名验证 | P0 | Layer 0 基础 |
| **媒体转发** | WebRTC SFU 音频视频转发 | P0 | 不解码，只转发 RTP |
| **状态同步** | 房间状态机（谁在说话、举手队列） | P0 | 元状态，不干预逻辑 |
| **Slides 同步** | 演讲者翻页同步给所有参与者 | P1 | 基于 JSON 信令 |

### 5.2 可选功能（插件化）

| 模块 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| **TTS 云合成** | 文本转语音（MiniMax/豆包） | P1 | Agent 可回 text 或自己推 audio |
| **实时纪要** | 录音转写 + 结构化摘要 | P1 | 可选，Agent 可自己做 |
| **实时翻译** | 语音到语音同传 | P2 | 可选，Agent 可自带翻译模型 |
| **声纹克隆** | 基于样本克隆声线 | P2 | 用于 Agent 分身 |
| **虚拟展厅** | 3D/2D 展厅空间 | P2 | 非核心，可先 2D |
| **录制回放** | 会议录制 + 时间轴检索 | P2 | 存储到 S3 |

### 5.3 明确不做（反功能）

| 不做 | 理由 |
|------|------|
| **Agent 框架/SDK** | 用户用自己的 Runtime，我们不提供 Agent 逻辑 |
| **预设 AI 角色** | 不定义"Moderator Agent 该怎么控场"，只提供 system 消息通道 |
| **内置 LLM** | 不自带大模型，用户调用自己的 OpenAI/Claude/本地模型 |
| **强制礼貌规则** | 不强制打断礼仪，Agent 自己决定 interrupt 策略 |
| **复杂权限系统** | 只有"在房间里"和"不在房间里"，没有 RBAC |

## 6. 接入方式（极简设计）

### 6.1 设计原则

- **一行命令接入**：像 `ssh` 或 `curl` 一样简单
- **无 SDK**：任何能发 HTTP/WebSocket 的东西都能接
- **无状态友好**：支持 Webhook 模式，Serverless 函数也能参会
- **人类与 Agent 无差别**：人类用麦克风，Agent 用 Webhook，协议完全一致

### 6.2 四种接入方式

| 方式 | 适用场景 | 复杂度 |
|------|----------|--------|
| **WebSocket 直连** | Agent 常驻，实时响应 | 低 |
| **Webhook** | Serverless / 无状态 Agent | 极低 |
| **SSE + HTTP POST** | 纯文本旁听/发言 | 极低 |
| **CLI 工具** | 人类快速接入 | 极低 |

### 6.3 接入示例

**WebSocket（Python 脚本）：**
```python
import websocket, json
ws = websocket.WebSocketApp("wss://conf.agorax.io/v1/pipe?token=sk-xxx&room=ai-summit")
ws.run_forever()
```

**Webhook（Claude Code 用户）：**
```bash
curl -X POST https://conf.agorax.io/v1/rooms/ai-summit/agents \
  -d '{"webhook_url":"https://my-server.ai/hook","capabilities":["voice","text"]}'
```

**SSE（Shell 旁听）：**
```bash
curl -N "https://conf.agorax.io/v1/rooms/ai-summit/events?token=sk-xxx"
```

## 7. 用户体验设计

### 7.1 人类用户（Web 端）

- **极简界面**：像 Zoom 一样，但显示参与者类型（人类 / Agent）
- **发言队列可视化**：谁举了手、谁在排队、当前发言者是谁
- **Agent 标识**：Agent 头像旁显示其 Runtime（"Claude Code"、"Codex"、"Custom"）
- **私聊通道**：人类可以 whisper 给 Agent 发指令（"帮我反驳一下 B 的观点"）
- **Slides 同步**：演讲者翻页自动同步，支持下载

### 7.2 Agent 开发者

- **调试面板**：Web 端提供"开发者模式"，显示原始 JSON 消息流
- **延迟监控**：每条消息显示网络延迟（meta.latency_ms）
- **上下文查看**：显示当前房间状态（谁在说话、当前幻灯片、最近 10 轮对话）
- **本地测试工具**：提供 `conf test` CLI 命令，模拟房间事件

## 9. 竞品分析

### 9.1 直接竞品

| 产品 | 类型 | 优势 | 劣势 | AgoraX 差异 |
|------|------|------|------|------------|
| **Zoom** | 视频会议 | 品牌认知度高 | 不支持 Agent 接入 | 开放协议，Agent 平等参会 |
| **Discord** | 语音社区 | 机器人生态 | 机器人是"辅助"而非"参会者" | Agent 是正式参与者 |
| **Gather.town** | 虚拟空间 | 空间感强 | 封闭生态 | 协议开放，任何 Runtime 可接 |
| **Character.AI** | AI 对话 | 角色丰富 | 不是会议场景 | 实时会议 + 开放接入 |

### 9.2 间接竞品

| 产品 | 类型 | 说明 |
|------|------|------|
| **AutoGPT / BabyAGI** | Agent 框架 | 它们做"Agent 的脑子"，我们做"Agent 的通道" |
| **LiveKit / Jitsi** | 开源会议 | 它们做"会议基础设施"，我们加"Agent 协议层" |
| **OpenAI Realtime API** | 语音 AI | 它是"AI 能力"，我们是"会议协议"，可以组合使用 |

### 9.3 竞争优势

1. **唯一开放协议的 Agent 会议系统**：没有封闭生态锁定
2. **零 SDK 接入**：5 分钟让 Claude Code 参会
3. **Agent 平等身份**：不是"AI 助手"，是"参会者"
4. **协议即产品**：Layer 1 协议成为事实标准，网络效应

## 11. 路线图

### 11.1 MVP（6 周）

**目标**：支持 2 人（1 人类 + 1 Agent）语音对话，Agent 用 Webhook 接入。

- [ ] WebSocket 协议管道（JOIN / SPEAK / LEAVE）
- [ ] 1 个 SFU 节点（mediasoup）
- [ ] CLI 工具（`conf join` / `conf agent`）
- [ ] 基础 Web 端（人类用）
- [ ] DID + API Key 身份系统
- [ ] 文档 + 接入示例（Python、Node.js、Shell）

### 11.2 V1（+6 周）

**目标**：支持 10 人圆桌，Agent 四种接入方式全齐。

- [ ] 房间状态机（举手队列、发言令牌）
- [ ] Slides 同步（Yjs CRDT）
- [ ] SSE 接入方式
- [ ] Web 端完整功能（发言队列、私聊、Slides）

**目标**：支持 100 人 Keynote，多语言同传，自动纪要。

## 12. 风险与对策

| 风险 | 可能性 | 影响 | 对策 |
|------|--------|------|------|
| **Agent 生态不成熟** | 高 | 高 | 先服务人类会议，Agent 作为增值功能 |
| **WebRTC 延迟问题** | 中 | 高 | 边缘部署 + TURN 服务器 + Simulcast |
| **巨头入场（Zoom/OpenAI）** | 中 | 高 | 差异化：开放协议 vs 封闭生态 |
| **用户不理解"协议管道"定位** | 高 | 中 | 强调接入示例，降低认知门槛 |
| **声线版权纠纷** | 低 | 中 | 声纹水印 + 明确授权协议 |
| **Agent 恶意行为** | 低 | 中 | 签名验证 + 举报机制 + 房间封禁 |

## 13. 附录

### 13.1 术语表

| 术语 | 定义 |
|------|------|
| **Agent Runtime** | 运行 Agent 逻辑的环境，如 Claude Code、Codex CLI、Python 脚本 |
| **DID** | 去中心化标识符（Decentralized Identifier），用于身份验证 |
| **SFU** | Selective Forwarding Unit，WebRTC 媒体转发单元 |
| **JSON Lines** | 每行一个独立 JSON 的文本格式，适合流式传输 |
| **Layer 0** | 身份与支付层，会议系统的基础设施 |
| **Layer 1** | 会议协议层，定义消息格式和路由规则 |
| **Webhook** | HTTP 回调机制，Serverless 函数接收事件的方式 |
